# Phase 18: Content Quality Pipeline

## Overview

Extend the content quality system from 13 deterministic regex checks to a multi-tier pipeline that includes domain-specific bible checks, LLM-as-judge evaluation (GPT-5.4), and auto-rewrite for low-scoring content. All three content types covered: onboarding pages, blog posts, and Reddit comments.

**Decision history:** 12 research agents evaluated DeepEval, Guardrails AI, and Google Check Grounding. Key outcomes:
- **Guardrails AI framework:** REJECTED (12/12 agents). Pre-1.0, 25+ deps, REASK is a black box. We already use the underlying libraries directly in `guardrails_checks.py`.
- **DeepEval library:** REJECTED (8/12 agents). 7 transitive deps including telemetry (posthog, sentry-sdk). We DIY the G-Eval pattern in ~150-200 lines.
- **Google Check Grounding:** CUT from MVP (9/12 agents). NLI fails on negation, false positives on marketing copy, adds GCP as a vendor for one API call. Bible regex checks cover 80% of the value.
- **LLM Judge model:** GPT-5.4 ($2.50/1M input, $20/1M output). Cross-model evaluation avoids self-preference bias (Claude judging Claude inflates 10-15%). At our volume (~$0.035/article) cost is negligible.

## Architecture

```
  CONTENT GENERATED BY CLAUDE
           |
           v
  +================================================================+
  |                    QUALITY PIPELINE                             |
  |                                                                |
  |  STEP 1: Collect inputs                              FREE     |
  |  STEP 2: Tier 1 -- 13 deterministic checks           FREE     |
  |  STEP 3: Tier 1b -- Bible rule checks                FREE     |
  |           Short-circuit if critical                            |
  |  STEP 4: Tier 2 -- LLM Judge (GPT-5.4)              ~$0.035  |
  |  STEP 5: Merge results + compute score (0-100)               |
  |           Score >= 70 --> STEP 6: Store. Done.                |
  |           Score < 70  --> STEP 7: Auto-rewrite                |
  |  STEP 6: Store in qa_results JSONB                            |
  |                                                                |
  +================================================================+

  COST PER ARTICLE:
  Score >= 70 (no rewrite):    ~$0.035
  Score < 70 (with rewrite):   ~$0.090
  Average (est. 20% rewrite):  ~$0.046
```

## Step-by-Step Pipeline

### STEP 1: Collect the Inputs

Before any checks run, gather everything the pipeline needs:

- **fields** — The actual content Claude wrote (`meta_title`, `meta_description`, `top_description`, `bottom_description`, `faq_answers`)
- **brand_config** — Vocabulary rules (banned words, competitors, AI prompt snippet)
- **content_brief** — What the page should be about (primary keyword, LSI terms, target word count)
- **matched_bibles** — Domain-specific knowledge bibles matched via keyword overlap with the page topic

### STEP 2: Tier 1 -- Deterministic Checks (Existing 13)

Run existing regex/string-matching checks in `content_quality.py`. These are battle-tested:

| # | Check | What It Catches |
|---|-------|----------------|
| 1 | Tier 1 AI words | "leverage", "elevate", "delve" |
| 2 | Tier 2 AI words | "seamlessly", "tapestry", "testament" |
| 3 | AI phrases | "In today's world", "Whether you're a" |
| 4 | Banned words | Client-specific forbidden terms from brand_config |
| 5 | Competitor mentions | Never mention competitors by name |
| 6 | Em dash abuse | AI loves em dashes, humans don't |
| 7 | Colon-intro pattern | "Key Benefits:" lazy heading format |
| 8 | Exclamation marks | More than 1 per field |
| 9 | Sentence starters | >40% starting with same word |
| 10-13 | Blog-specific | Humanizer artifacts, internal links, etc. |

All return `list[QualityIssue]` with `confidence = 1.0` (definite problems). Free, <10ms.

### STEP 3: Tier 1b -- Bible Rule Checks (New)

If any bibles matched this page's topic, run their `qa_rules` against the content. Same regex/string pattern as Tier 1, but rules come from the bible instead of being hardcoded.

**Check types:**

| Check | Example Rule | How It Works |
|-------|-------------|-------------|
| `bible_preferred_term` | Use "needle grouping" not "needle configuration" | Regex for the wrong term |
| `bible_banned_claim` | Never say "only brand to offer membranes" | Fuzzy match on claim + context in same sentence |
| `bible_wrong_attribution` | "membrane" belongs to "cartridge needle", not "tattoo pen" | Feature term + wrong component in same sentence |
| `bible_term_context` | "membrane" should appear with "recoil protection" not "ink savings" | Same-sentence co-occurrence check |

**Key design decision:** These go in `content_quality.py`, NOT a separate file. They're "banned words with a different data source" — same pattern, same code path, different input. Per the DX agent: "The developer who inherits this code needs to understand: regex patterns return quality issues. That is the core mental model."

Returns `list[QualityIssue]` with `confidence = 1.0`. Free, <10ms.

### SHORT-CIRCUIT DECISION

Before spending money on Tier 2, check if there are critical Tier 1 failures:

- `tier1_ai_word` (content literally says "delve" — no LLM needed)
- `bible_banned_claim` (factually wrong claim the domain expert flagged)

If critical failures exist, skip Tier 2. Why pay for an LLM opinion when we already know it needs fixing?

### STEP 4: Tier 2 -- LLM Judge (GPT-5.4)

Send content to GPT-5.4 with structured evaluation rubrics. Catches things regex can't — subjective quality, tone, coherence, brief adherence.

**Three checks:**

#### Naturalness Scoring
```
Prompt: "Rate this text 0-1:
  1.0 = human expert wrote it
  0.7 = mostly natural
  0.4 = noticeably AI
  0.0 = obvious AI slop
  Return {score, reasoning}"
```
Why: Catches overall AI "feel" that no single regex can detect. Text might pass all 13 deterministic checks but still feel like AI wrote it.

#### Brief Adherence
```
Prompt: "Given this content brief:
  - Primary keyword: X
  - Must cover: [topics]
  - Target: 1500 words
  Does the content address the brief? Score 0-1.
  List any brief points missed."
```
Why: Claude sometimes goes off on tangents. Brief says "cover needle groupings" but Claude writes 500 words about tattoo aftercare instead.

#### Heading Structure
```
Prompt: "Do the headings flow logically?
  Is the keyword in H1?
  Do H2s support the topic?"
```
Why: SEO-critical. Regex can check "is keyword in H1" but can't judge whether heading hierarchy makes logical sense.

**Implementation:** DIY via direct `httpx` calls to OpenAI API. ~150-200 lines in `llm_judge.py`. No DeepEval, no framework — just an HTTP call with a rubric prompt.

**Results are ADVISORY — never blocking.** LLM scores vary +/-10-15% between runs. Shown as yellow warnings, not red blockers. The operator decides. Returns `list[QualityIssue]` with `confidence = 0.0-1.0`.

**Why GPT-5.4 not Claude:** Self-preference bias. Research shows LLMs rate their own output 10-15% higher. Claude wrote the content — different model family gives honest evaluation.

~$0.035 per article, ~2s latency.

### STEP 5: Merge Results + Compute Score

Combine all issues from all tiers into one list. Compute aggregate quality score (0-100):

**Scoring formula:**
```
Start at 100
- Each Tier 1 error:     -5 points
- Each Tier 1 warning:   -2 points
- Each bible violation:  -5 points
- Naturalness < 0.6:     -10 points
- Brief adherence < 0.7: -10 points
Floor at 0
```

**Score tiers:**
| Range | Label | Color |
|-------|-------|-------|
| 90-100 | Publish Ready | Green |
| 70-89 | Minor Issues | Yellow |
| 50-69 | Needs Attention | Orange |
| 0-49 | Needs Rewrite | Red |

### STEP 6: Store in qa_results JSONB

Write to existing `qa_results` column on `PageContent` / `BlogPost`. No migration needed — JSONB is schema-free.

```json
{
  "passed": true,
  "score": 88,
  "checked_at": "2026-03-06T...",
  "issues": [
    {
      "type": "tier1_ai_word",
      "field": "bottom_description",

    "brief_adherence": 0.88,
    "heading_structure": 0.92,
    "cost_usd": 0.035,
    "latency_ms": 1800
  },
  "bibles_matched": ["tattoo-cartridge-needles"]
}
```

### STEP 7: Auto-Rewrite (Score < 70)

When content scores below 70, automatically fix it. One attempt only.

#### 7a. Filter to Fixable Issues Only

Only send issues Claude can actually act on with surgical edits:

**FIXABLE (send to Claude):**
- `tier1_ai_word` — "Replace 'leverage'"
- `tier1_ai_phrase` — "Remove 'in today's world'"
- `em_dash_overuse` — "Reduce em dashes"
- `banned_word` — "Remove 'cheap'"
- `competitor_mention` — "Remove 'CompanyX'"
- `bible_preferred_term` — "Use 'needle grouping' not 'needle configuration'"
- `bible_banned_claim` — "Remove claim about 'only brand to offer'"
- `bible_wrong_attribution` — "Membrane is a cartridge feature, not a pen feature"
- `bible_term_context` — "Membrane + ink savings is wrong — rewrite to mention backflow prevention"

**NOT FIXABLE (report only):**
- `llm_naturalness` — subjective score, "be more natural" is too vague
- `llm_heading_structure` — restructuring headings changes the whole piece
- `sentence_starters` — rewriting openers risks changing meaning

#### 7b. Build Targeted Rewrite Prompt

NOT "rewrite this content" — that nukes SEO optimization. Instead: "make these specific edits."

```
System:
"You are a content editor. You will receive HTML content and a list of
specific issues. Fix ONLY the listed issues. Do not change anything else.
Preserve all HTML structure, headings, keywords, and formatting. Return
the full corrected content in the same format."

User:
"## Content to Fix
[field: top_description]
<h2>Why Leverage Cartridge Needles for Your Studio</h2>
<p>In today's world, the membrane technology in tattoo pens saves ink...</p>

## Issues to Fix
1. [top_description] AI word 'leverage' — replace with natural alternative
2. [top_description] AI phrase 'In today's world' — remove or rephrase
3. [top_description] Membrane attributed to 'tattoo pen' — membrane is a
   cartridge needle feature
4. [top_description] 'saves ink' used with membrane — membranes prevent
   backflow for hygiene, they don't save ink

Fix each issue. Change nothing else."
```

#### 7c. Send to Claude

Same Claude model and `ClaudeClient` as generation. One API call per field that has issues (batch all issues for a field into one call).

```
Issues grouped by field:

top_description: 4 issues  --> 1 Claude call
bottom_description: 1 issue --> 1 Claude call
faq_answers: 0 issues       --> skip

Total: 2 API calls (~$0.01-0.02)
Latency: ~3-5 seconds (parallel)
```

#### 7d. Re-run Quality Pipeline on Fixed Content

Run Steps 2-5 again on the new content to verify:
- Did the fix actually resolve the issues?
- Did the fix accidentally introduce NEW issues? (e.g., Claude replaces "leverage" with "utilize" — another AI word)

#### 7e. Keep the Better Version

```
IF fixed_score > original_score:
    Use fixed content
    Store both versions

ELSE:
    Keep original content (fix made it worse)
    Flag for manual review

HARD LIMIT: 1 rewrite attempt only.
No cascading retries. If one fix doesn't get it above 70,
it goes to the operator.
```

**Why max 1 retry:** Each retry drifts further from original intent. One targeted fix pass handles 80%+ of cases. If content is so bad surgical fixes can't save it, a human needs to look at it.

## Vertical Knowledge Bible System

### Data Model: `vertical_bibles` table

| Column | Type | Purpose |
|--------|------|---------|
| `id` | UUID PK | Standard |
| `project_id` | FK -> projects | Per-project bibles |
| `name` | String(255) | "Tattoo Cartridge Needles" |
| `slug` | String(255), unique w/ project | "tattoo-cartridge-needles" |
| `content_md` | Text | Full markdown knowledge doc (injected into prompts) |
| `trigger_keywords` | JSONB array | Keywords that activate this bible |
| `qa_rules` | JSONB dict | Structured rules for quality checks |
| `sort_order` | Integer | Priority when multiple bibles match |
| `is_active` | Boolean | Enable/disable without deleting |
| `created_at`, `updated_at` | Timestamps | Standard |

Unique constraint on `(project_id, slug)`.

### qa_rules JSON Structure

```json
{
  "term_context_rules": [
    {
      "term": "membrane",
      "correct_context": ["recoil", "machine protection", "health compliance"],
      "wrong_contexts": ["ink savings", "ink efficiency"],
      "explanation": "Membranes prevent backflow, they don't save ink"
    }
  ],
  "feature_attribution": [
    {
      "feature": "membrane",
      "correct_component": "cartridge needle",
      "wrong_components": ["tattoo pen", "tattoo ink"]
    }
  ],
  "banned_claims": [
    {
      "claim": "only brand to offer",
      "context": "membrane",
      "reason": "All major cartridge brands include membranes - industry standard"
    }
  ],
  "preferred_terms": [
    {
      "use": "needle grouping",
      "instead_of": "needle configuration"
    }
  ]
}
```

### Bible Markdown Schema

```markdown
---
name: Tattoo Cartridge Needles
slug: tattoo-cartridge-needles
trigger_keywords:
  - cartridge needle
  - membrane
  - needle grouping
  - round liner
  - taper
---

## Domain Overview
Cartridge needles are pre-assembled needle modules that snap into a pen...

## Correct Terminology
| Use This | Not This | Why |
|----------|----------|-----|
| needle grouping | needle configuration | Industry standard |

## Feature-to-Benefit Mapping
| Feature | Benefit | How to Write It |
|---------|---------|-----------------|
| Built-in membrane | Prevents cross-contamination | "Each cartridge has a built-in membrane..." |

## What NOT to Say
- "Membrane saves ink" -- prevents backflow, doesn't reduce consumption
- "Only [brand] offers membrane cartridges" -- industry standard

## Component Relationships
- Cartridge needles go INTO tattoo pens
- Needles are SEPARATE from inks -- never imply needle features affect ink
```

### Bible Matching (Deterministic)

Word-boundary substring match of `trigger_keywords` against page's `primary_keyword`. No LLM call, no embeddings. ~50 bibles x ~10 keywords = trivially fast.

### Prompt Injection

New `## Domain Knowledge` section in the user prompt, between `## SEO Targets` and `## Brand Voice`. Concatenates matched bibles' `content_md` with a `max_chars` guard (~8000 chars).

```
POST   /projects/{id}/bibles                          -- Create bible

PUT    /projects/{id}/bibles/{bid}                    -- Update bible

POST   /projects/{id}/bibles/import                   -- Import from .md
GET    /projects/{id}/bibles/{bid}/export              -- Export as .md

```

### Transcript -> Bible Generation (Phase 2)

`POST /projects/{id}/bibles/generate-from-transcript`

1. Send transcript to Claude with structured extraction prompt
2. Claude returns draft bible (content_md, trigger_keywords, qa_rules)
3. Return draft to frontend for human review/editing before saving
4. User reviews, edits, confirms -> saved as VerticalBible

Draft-first, not auto-save. The AI generates a draft that the user reviews.
